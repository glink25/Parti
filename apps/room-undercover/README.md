# 谁是卧底房间设计说明

本文档记录 `room-undercover` 当前版本的产品设计、游戏规则、状态模型和实现边界，作为后续迭代的设计基线。若文档与代码不一致，以当前版本代码为准，并应在修改代码时同步更新本文档。

## 1. 产品定位

“谁是卧底 · 发牌房”是一个轻量聚会游戏工具，负责选词、分配身份、私密发牌、记录出局和揭晓答案，不负责语音讨论、投票、计时或计分。

产品提供两个入口：

- **联机模式**：每名玩家使用自己的设备加入 Parti 房间，系统向每台设备私密发送词牌。
- **牌盒模式**：只使用一台设备，本地生成带序号的牌盒，玩家轮流查看自己的词牌。该模式与房间状态完全独立，不需要其他玩家联网。

支持人数为 3–12 人，不允许旁观者加入。

## 2. 通用游戏规则

### 2.1 身份与词牌

每轮玩家分为两种隐藏身份：

- `civilian`：平民，获得同一个平民词。
- `undercover`：卧底，获得另一个相近词；空白牌玩法中获得空白词。

发牌时会先随机打乱玩家，再按人数取前 N 名作为卧底：

| 收牌人数 | 卧底人数 |
| --- | ---: |
| 3–5 | 1 |
| 6–9 | 2 |
| 10–12 | 3 |

经典词库中的一对词没有永久固定的平民/卧底方向。每次发牌时有 50% 概率交换两词方向，避免玩家通过词语本身推断身份。自定义词语不交换方向。

### 2.2 回合流程

1. 选择参与人数或等待玩家加入。
2. 选择词牌玩法与分类。
3. 洗牌并发牌。
4. 玩家私下查看并记住自己的词语。
5. 玩家在线下依次描述、讨论和投票。
6. 被投出的玩家在界面中确认出局。
7. 若仍有卧底在场，游戏继续；若所有卧底均已出局，本轮结束并揭晓平民词与卧底词。

当前版本只实现“所有卧底出局”的结束条件，不判断平民人数与卧底人数相等等其他胜负条件，也不记录胜方或分数。

### 2.3 词牌分类

内置词库位于 `src/words.ts`，每个分类当前包含 25 对词：

- `entertainment`：娱乐，影视、音乐、游戏等。
- `weapons`：武器，古今装备与防具。
- `daily`：日常，生活中的相似事物。
- `nsfw`：含蓄成人向内容，标记为 18+。

至少必须选择一个分类。系统从所有已选分类中随机取词对，并在当前运行周期内尽量避免重复；候选词全部用完后清空该候选集合的使用记录并重新开始。

## 3. 联机模式

### 3.1 参与方式

所有玩家加入同一个 Parti 房间。房主也是玩家之一，但在主持玩法中不收牌。房间使用快照同步公开状态，使用定向事件发送私人词牌。

界面以首屏完整展示为目标：

- 左侧或上方是当前玩家的核心词牌区域。
- 右侧或下方是紧凑玩家列表和本轮状态。
- 房主设置收纳在“房主设置”弹窗内，避免控制项持续占用页面高度。
- 移动端改为单列布局，玩家列表采用紧凑网格并限制高度。

### 3.2 房主发牌玩法

房主设置弹窗提供三种玩法：

#### 经典词库 `classic`

- 房主与其他玩家都参与并收牌。
- 房主选择一个或多个内置词牌分类。
- 至少需要 3 名在场玩家。

#### 空白牌 `blank`

- 房主只担任主持，不收牌。
- 至少需要 3 名非房主玩家。
- 从所选词对中随机取一个词作为平民词，卧底收到空白牌。

#### 自定义 `custom`

- 房主只担任主持，不收牌。
- 房主填写明确的平民词和卧底词。
- 两个词会去除首尾空白，必须非空、互不相同，且各自不超过 20 个字符。
- 自定义词的方向固定，不进行随机交换。

### 3.3 私密发牌

角色和全部词牌只保存在 Worker 模块内存的 `currentCards` 中，不写入公开房间状态。

发牌后 Worker 通过 `undercover:card` 定向事件向每名收牌玩家发送：

```ts
{ round: number; word: string }
```

事件不包含 `role`，因此客户端只能看到自己的词，不知道自己的阵营和其他人的词。

玩家收到牌后先看到密封状态，主动点击才能翻牌；点击“我记住了”后重新隐藏词牌。

### 3.4 出局与结算

收牌玩家被线下投出后，点击“我被投出局了”，并在确认弹窗中提交 `round:eliminateSelf`。

Worker 会校验：

- 当前回合处于 `active`。
- 操作者仍在房间中且本轮收到过牌。
- 操作者有对应隐藏牌。
- 操作者尚未出局。

出局身份不会立即公开。每次出局后调用统一的 `resolveElimination`：

- 只要有一名卧底未出局，回合保持 `active`。
- 所有卧底出局后，回合进入 `finished`，公开平民词和卧底词。

### 3.5 加入、离开与恢复

- 中途加入且错过发牌的玩家等待下一轮。
- 玩家离开时会从已发牌玩家和公开玩家列表中移除。
- 断线重连时，如果 Worker 内存仍有该玩家的牌，会重新向其发送私人词牌。
- 房主运行环境从持久化快照恢复时，隐藏牌无法从公开快照恢复，因此当前轮次会重置为等待状态，并提示房主重新发牌。

本房间只实现当前版本的数据结构，不设计跨版本房间数据迁移。

## 4. 牌盒模式

### 4.1 设计目的

牌盒模式用于“不是每个人都有手机”的线下场景。所有逻辑在当前浏览器页面本地运行，不调用房间 action，也不修改联机模式的权威状态。

牌盒模式当前只使用经典内置词库，不提供空白牌或自定义词玩法。

### 4.2 设置与发牌

- 可选择 3–12 名玩家。
- 使用与联机模式相同的分类按钮和内置词库。
- 点击“洗牌并发牌”后生成从 1 开始的连续序号卡牌。
- 每张本地卡保存序号、隐藏角色、词语、是否看过和是否出局。
- 修改人数或分类会清空当前牌盒与结算状态。
- 重新发牌会重置所有查看、出局和结算状态。

### 4.3 查看词牌

首次点击未查看卡牌时直接打开私密词牌弹窗。玩家需要同时记住序号和词语，然后点击“我记住了”：

- 弹窗关闭。
- 卡牌标记为“已看过”。
- 卡牌网格仍只显示序号，不显示词语或阵营。

已看过的卡牌再次点击时不会直接展示词语，而是提供两个动作：

- **忘了，再看一次**：再次打开该玩家的私密词牌。
- **被投出局了**：将该序号标记为出局并锁定卡牌，不能撤销或重复操作。

### 4.4 本地结算

牌盒模式保留隐藏角色仅用于本地判断，不在牌面网格中公开。点击“被投出局了”后，将本地牌数据投影为与联机模式一致的卡牌记录，并复用 `resolveElimination`：

- 卧底尚未全部出局：关闭操作弹窗，保留牌盒，游戏继续。
- 所有卧底出局：牌盒区域切换为结算页，展示平民词与卧底词。

结算后不能继续操作旧卡牌；点击“重新洗牌并发牌”开启新一轮。

## 5. 状态模型

### 5.1 联机公开状态 `RoomState`

```ts
type RoomState = {
  phase: 'waiting' | 'active' | 'finished';
  hostId: string | null;
  players: PublicPlayer[];
  selectedMode: 'classic' | 'blank' | 'custom';
  roundMode: 'classic' | 'blank' | 'custom' | null;
  selectedCategories: Category[];
  round: number;
  dealtPlayerIds: string[];
  eliminatedPlayerIds: string[];
  revealedWords: { civilian: string; undercover: string } | null;
  notice: string | null;
};
```

公开状态可以包含玩家、设置、进度和结算结果，但回合进行期间不得包含角色或秘密词牌。

### 5.2 联机隐藏状态

```ts
let currentCards: Record<string, {
  role: 'civilian' | 'undercover';
  word: string;
}>;
```

`currentCards` 只存在于 Worker 模块内存。`usedPairIds` 同样是运行期内存，用于减少重复词对。

### 5.3 牌盒本地状态

```ts
type ShareCard = {
  number: number;
  word: string;
  role: 'civilian' | 'undercover';
  seen: boolean;
  eliminated: boolean;
};
```

虽然代码内部沿用 `share` 命名，本地模式的产品名称统一为“牌盒模式”。这些数据只存在于当前 React 页面内存中，刷新页面会丢失。

## 6. 房间协议

### 6.1 Actions

| Action | 权限 | 作用 |
| --- | --- | --- |
| `settings:setMode` | 房主 | 设置 `classic`、`blank` 或 `custom` |
| `settings:setCategories` | 房主 | 设置至少一个有效词牌分类 |
| `round:deal` | 房主 | 校验设置、洗牌、发牌并开启新一轮 |
| `round:eliminateSelf` | 本轮收牌玩家 | 确认自己出局并触发结算判断 |

### 6.2 私人事件

| Event | 接收者 | Payload |
| --- | --- | --- |
| `undercover:card` | 单个收牌玩家 | `{ round, word }` |

## 7. 视觉与交互原则

- 视觉主题是深紫色牌桌、金色主操作和低饱和状态色。
- 核心秘密信息使用大字号牌面呈现，普通状态信息保持紧凑。
- 尽量让主要操作和状态在首屏完整出现，避免依赖纵向滚动。
- 危险或不可撤销操作必须经过确认弹窗。
- 已查看、已出局、等待和完成状态同时使用文字与颜色表达，不只依赖颜色。
- 弹窗支持 `Escape` 关闭；所有主要操作使用原生按钮并保留键盘焦点样式。
- 遵循 `prefers-reduced-motion`，减少不必要动画。

## 9. 验证方式

常用命令：

```bash
pnpm --filter @parti/room-undercover typecheck
pnpm --filter @parti/room-undercover test
pnpm --filter @parti/room-undercover build
```

自动验证范围：

- TypeScript 类型正确性。
- 词牌分类与自定义词校验。
- 不重复选词逻辑。
- 洗牌、卧底人数和空白牌分配。
- 私人牌 payload 不泄露角色。
- 联机模式和牌盒模式共用的出局结算规则。
- Vite 生产构建和 Worker 合约。

自动化测试不覆盖 UI、动画、弹窗交互、多人网络流程或实际游戏手感。集成行为与移动端首屏布局应由真人玩家在目标设备上试玩确认。

## 10. 当前明确不做的内容

- 应用内讨论、语音、计时和自动轮流描述。
- 应用内投票或代替其他玩家确认出局。
- 撤销出局。
- 平民被淘汰到特定人数后的胜负判断。
- 计分、连胜、历史战绩或跨刷新牌局保存。
- 牌盒模式联机同步。
- 跨版本房间数据迁移或兼容层。
