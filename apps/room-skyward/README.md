# 云端远征（Skyward）

云端远征是一个运行在 Parti Room Runtime 中的 1–4 人合作垂直攀登游戏。它以《涂鸦跳跃》式的自动弹跳为基础，加入自由分散、共享世界、远程队友感知、射击、踩踏、跨高度机关、周期 Boss 和团队全灭规则。

项目不仅是一套关卡，也是一套可持续扩充的内容框架。普通区域、协作区域和 Boss 区域使用同一个地形模块系统；新增地形、生态、敌人、Buff 或 Boss 时，不应修改世界生成主流程。

## 设计理念

### 人都在，游戏才在

云端远征首先是一款同场聚会游戏。玩家可以在世界中相隔很远，各自探索和战斗，但整支队伍共享一次远征：

- 所有人准备后才开始，单人也可以开局。
- 开始后不接受中途加入。
- 任何一名在线存活玩家都能维持远征；全员同时死亡则立即结束。
- 掉线玩家的席位保留，不再算作存活，也会阻塞下一次 Boss 集结。
- 房主负责权威 Worker；房主关闭属于本局 panic，不实现主机迁移。
- 团灭后只有房主可以让在线成员返回准备阶段。

这套规则有意把“共同在场”变成游戏约束，而不是试图隐藏聚会成员离开的事实。

### 自由分散，但必须重聚

玩家拥有独立镜头，可以处在完全不同的高度。留在低处的玩家能够替团队维持更低的虚空线，为上方队友保留更大的坠落空间。这是一种合法策略。

每个区段末尾的 Boss 门负责重新收拢队伍。所有本局成员必须亲自抵达集合高度且处于在线、存活状态，Boss 才会开始。Boss 因而同时承担：

- 合作战斗高潮；
- 对每位玩家个人攀登能力的检查；
- 团队重新集合；
- 抬升虚空线和回收旧世界的逻辑边界；
- 下一个生态区段的入口。

### 网络延迟不能控制本地手感

移动和弹跳由每个客户端立即模拟。客户端不会等待房主回包才移动，否则 PeerJS 延迟会直接变成操作延迟。

房主不模拟或纠正玩家的普通运动，而是充当共享事实的排序器和广播站。它负责会影响共同结果的内容：

- 开局与玩家名单；
- 玩家公开位置摘要与团队虚空线；
- 伤害、死亡、复活与无敌时间；
- 敌人是否已被击败；
- Buff 是否已领取；
- 协作机关状态；
- Boss 集结、生命值与击败结果；
- 本局成绩与房间最高纪录。

玩家之间没有实体碰撞。本人位置始终由本地模拟决定；远端角色的插值不会把本地玩家从平台上撞下去。

## 当前玩法

### 操作

- 角色持续自动弹跳。
- 触屏左右半区控制横向移动。
- 桌面端支持方向键和 `A` / `D`。
- 固定开火按钮；桌面端可使用空格。
- 支持方向传感器的设备可以主动开启重力感应。
- 只有玩家自身会在世界左右边缘环绕。平台、怪物、Buff和子弹目标在边缘处截断，不会在另一侧产生不可见碰撞副本。

游戏使用固定 `900 × 1600`（9:16）逻辑视野。不同屏幕只进行等比缩放；多余区域用于背景和非关键提示，不能让大屏设备看到更多关卡。

### 战斗与 Buff

- 普通子弹垂直向上发射，并受房主射速冷却校验。
- 玩家下落时接触怪物顶部可以踩踏消灭怪物并重新弹起。
- 其他方向接触怪物会受到伤害。
- 复活后获得 2.2 秒房主权威无敌时间。
- 房主授权射击并广播给所有玩家；远端子弹只用于展示，没有友军伤害，也不会在观察者客户端提交命中。
- 射手客户端计算子弹轨迹和命中，再用授权 `shotId` 请求房主结算一次共享结果。

踩踏使用相同的客户端结果信任模型：客户端命中后立即反弹并乐观隐藏怪物，房主只验证敌人 ID、按到达顺序去重并把死亡结果写入 `defeatedEnemies`。它不再用延迟的玩家遥测位置否决踩踏。

当前 Buff：

| ID | 归属 | 效果 |
| --- | --- | --- |
| `rapid` | 个人 | 缩短射击冷却 |
| `spread` | 个人 | 额外显示和发射两侧散弹 |
| `power` | 个人 | 对 Boss 造成更高伤害，并改变子弹颜色 |
| `team-shield` | 团队 | 抵挡下一次普通怪物伤害 |

### 死亡、镜头与虚空

客户端维护两条不同的高度基准，不能合并：

1. `cameraBottom` 是玩家的不可回退进度底边，会发送给房主，用于计算团队虚空线。
2. `viewBottom` 只是本地实际镜头位置，可以在角色下坠时向下追踪。

团队虚空线是所有在线存活玩家 `cameraBottom` 的最小值，并且整体只会上升。玩家低于团队虚空线 60 世界单位后，由房主判定坠落死亡。

将两条基准拆开非常重要：如果实际镜头也被设计为只能上升，玩家向低处队友区域坠落时会离开自己的画面；如果权威进度底边可以下降，玩家又能通过持续坠落无限拖低虚空。

死亡后：

- 等待 4 秒；
- 若已经抵达当前 Boss 门，则在门区复活；
- 否则在个人进度底边和当前团队虚空线之上的安全高度复活；
- 获得 2.2 秒无敌；
- 如果等待期间所有在线玩家都死亡，远征立即结束，不再执行复活。

## 技术架构

```text
LittleJS 客户端（每位玩家一份）
  ├─ 输入、自动弹跳与本地即时物理
  ├─ 固定 9:16 镜头和下坠追踪
  ├─ 根据 seed 确定性生成静态地图
  ├─ 远端角色插值和远端子弹展示
  └─ parti.action 提交意图/遥测
                    │
                    ▼
Parti Host Runtime + room.worker
  ├─ 唯一权威 GameState
  ├─ 玩家名单、准备、掉线和重开
  ├─ 位置镜像、远端广播和团队虚空线
  ├─ 射击授权、命中去重、Buff、机关和 Boss 裁决
  ├─ 死亡、复活、无敌和全灭
  └─ 全量 snapshot + 瞬时 event
```

## 视觉素材注册

客户端视觉使用 `src/assets/catalog.ts` 作为唯一注册入口，场景代码只引用稳定素材 ID，不直接拼接文件路径。`AssetRegistry` 负责图片预加载、缓存和失败回退；生态、角色、敌人和 UI 分别通过 `registerBiomeTheme`、`registerCharacterSkin`、`registerEnemySkin` 和 `registerUiTheme` 注册。

投射物同样通过注册表管理。普通、散射和强力弹分别配置图片、世界单位尺寸和光晕；`rapid` 只影响射击频率。新增弹种时不得在场景渲染中直接拼接素材路径。

新增角色皮肤时，将尺寸一致的透明 PNG 放入 `public/assets/skyward/characters/`，注册 idle、两帧 walk、jump、hit 和 avatar 六类素材，再添加一个 `CharacterSkin`。新增生态时，准备背景以及平台 left/middle/right 三段素材，注册 `BiomeTheme`，并使主题 ID 与 `Biome.id` 一致。素材文件统一使用小写短横线命名；渲染器会自动收集注册依赖并预加载。

仓库中的正式素材来自 Kenney New Platformer Pack，许可见 `public/assets/skyward/LICENSE.txt`。本地原始素材包目录已被忽略，不应作为运行时依赖。

Worker 源码可以按模块开发，但 Parti 最终只加载一个 Worker 入口。Vite 插件使用 esbuild 将 `src/worker/index.ts` 及其共享依赖打包为 `worker.js`，同时把 `@parti/worker-sdk` 保持为外部依赖，交由 Runtime 注入。

## 地形系统

### 世界层次

地形系统分为四层：

#### Biome

`Biome` 定义一个生态主题的公共内容和视觉参数：

- 唯一 ID 和名称；
- 天空、雾气和平台颜色；
- 可用敌人类型；
- 后续可扩展主题音乐、装饰池、机关池和难度修正。

当前生态为晨光浮岛、苍翠天庭和雷鸣高塔。每击败一个 Boss，区段会进入下一个生态并持续提高难度；生态列表循环使用，但区段难度继续增长。

#### TerrainModule

`TerrainModule` 是约一屏高的代码化地形模块。模块包含：

- 稳定 `id`；
- 检索标签；
- 随机权重；
- 可出现的区域类型；
- 难度和人数范围；
- `generate(context)` 生成函数。

生成上下文提供当前 seed、区块索引、绝对高度、开局人数、难度、Biome、区域类型、入口连接器以及模块专用 RNG。

生成函数返回：

- 出口连接器；
- 平台；
- 敌人出生点；
- Buff 出生点；
- 后续可加入的触发区和动态行为描述。

每个区块还必须提供按上升顺序排列的 `route`。相邻区块共享确定性边界连接器，路线中的相邻平台必须通过当前角色物理模型的可达性检查；Boss 出口平台只有在对应 Boss 被击败后才启用。

模块必须使用上下文提供的 RNG，不能调用 `Math.random()`、当前时间或依赖全局生成顺序。

当前模块：

| 模块 | 区域 | 说明 |
| --- | --- | --- |
| `open-steps` | 普通 | 开放式随机台阶、敌人与可选 Buff |
| `wrap-zigzag` | 普通 | 强调左右环绕的折返路线 |
| `relay-spires` | 协作 | 跨高度限时接力桥，单人局使用更宽平台和更长时间 |
| `boss-gate` | Boss | 通向整屏宽集合平台的固定结构与等待守卫 |

#### RegionRecipe

区域配方目前由 `regionKindFor(index)` 表达：

- 普通区块侧重移动和战斗；
- 每三个区块周期性插入协作区块；
- 每七个区块的最后一个区块固定为 Boss 门。

区域类型不是另一套生成器。所有区域都从同一个模块注册表筛选候选模块，再按稳定 ID 排序和权重选择。后续可以将当前函数扩展为显式 `RegionRecipe` 注册表，以声明更复杂的模块比例、必需标签和终止条件，而无需改变模块接口。

#### Encounter / Mechanic

敌人、拾取物和协作接力是注入地形的遭遇与机制。当前它们仍直接由模块返回；类型已经保持独立，后续可以把它们提取为各自注册表，使同一地形复用不同敌人组合或机关。

### 连接器

模块入口和出口使用 `Connector`：

```ts
type Connector = {
  minX: number;
  maxX: number;
  minY: number;
  maxY: number;
  wrap: true;
};
```

连接器描述合法进入/离开范围，而不是要求作者了解相邻模块。生成器负责选择兼容模块并建立过渡。当前移动模型始终允许左右环绕，模块通过入口附近的起始平台和出口附近的末端平台完成衔接。

新增复杂移动能力后，连接器应增加明确的能力标签，例如冲刺、破墙或多人保持开关；不要根据平台坐标隐式推断能力。

### 确定性随机

同一局中所有客户端和 Worker 必须生成同一静态世界。随机数按以下键派生：

```text
runSeed : chunkIndex : channel
```

当前通道包括：

- `module`：选择地形模块；
- `terrain`：生成模块内部平台、敌人和 Buff；
- `exit`：生成跨模块连接位置。

不同通道互不影响。向装饰通道增加一次随机调用，不应改变平台或敌人的结果。

候选模块在选择前按稳定 ID 排序，因此模块注册代码的排列顺序不会改变结果。加入一个新模块则允许旧 seed 在新版本产生不同世界，这是有意的产品决定：Skyward 不维护跨版本 seed 兼容、历史录像或旧对局回放。

### 静态与动态状态边界

静态地图几何不进入 Parti state。客户端和 Worker按 seed 随用随生成，避免全量 snapshot 携带无限地图。

只有被玩家改变的结果进入权威 state，例如：

- `defeatedEnemies`；
- `claimedPickups`；
- `activeRelays`；
- Boss 状态；
- 团队 Buff。

这条边界是地图可无限扩展的前提。若未来加入移动平台，其当前位置不能只靠各端墙钟动画；应由固定模拟时间锚点推导，或把必要的动态阶段交给 Worker 同步。

### 新增地形模块

1. 在 `TERRAIN_MODULES` 中注册唯一 ID。
2. 声明区域、标签、权重、难度和人数范围。
3. 只使用 `ctx.rng` 生成随机内容。
4. 以 `ctx.baseY` 为绝对高度原点。
5. 为所有平台、敌人和 Buff 构造稳定 ID，通常包含区块索引。
6. 返回与实际末端路线一致的出口连接器。
7. 人工验证 1–4 人、入口到出口、左右环绕和相邻连接。
8. 协作模块必须提供可完成的单人变体。

模块错误不会在运行时被安全地形替换。地形是核心游戏代码；错误应直接暴露并在发布前修复。

## 联机协议设计

### 权威 State

`GameState` 主要包含：

- 阶段：`lobby`、`running`、`boss`、`gameover`；
- 房主 ID、隐藏 seed 和开局玩家名单；
- 玩家公开状态；
- 团队虚空线、最高高度和 Boss 数量；
- 当前 Boss 与下一扇门；
- 已击败敌人、已领取 Buff、团队 Buff 和活动接力桥；
- 房间内最高纪录；
- 当前提示信息。

Parti 当前采用全量 snapshot，因此 state 必须保持紧凑。静态平台、客户端子弹轨迹、粒子和装饰不应放入 state。

### Actions

| Action | 用途 |
| --- | --- |
| `setReady` | 设置准备状态；全员准备后开局 |
| `restart` | 团灭后仅房主返回准备阶段 |
| `move` | 横移方向变化，值为 `-1 / 0 / 1` |
| `telemetry` | 低频上报位置、速度和不可回退镜头底边 |
| `shoot` | 携带唯一 `shotId` 和本地起点请求房主授权开火 |
| `resolveHit` | 射手提交已授权弹丸对敌人或 Boss 的命中 |
| `stomp` | 提交踩踏敌人 |
| `claimPickup` | 请求领取确定性生成的 Buff |
| `activateRelay` | 激活协作区限时桥 |
| `reportDeath` | 客户端即时死亡后报告 `void` 或 `enemy` 原因 |
| `enableTilt` | 公开玩家是否启用重力操作 |

客户端不得把 action Promise 的成功当作游戏裁决成功；最终结果以随后的 state 或 event 为准。

### Events

| Event | 用途 |
| --- | --- |
| `skyward:start` | 开局瞬时反馈 |
| `skyward:shot` | 向其他客户端展示权威接受的远端子弹 |
| `skyward:hit` | 命中表现 |
| `skyward:pickup` | Buff 拾取提示 |
| `skyward:shield` | 团队护盾抵挡提示 |
| `skyward:death` | 死亡提示 |
| `skyward:boss` | Boss 开始 |
| `skyward:boss-defeated` | Boss 击败表现 |
| `skyward:gameover` | 结算瞬时表现 |

事件只用于可以错过的表现。结算、Boss生命、死亡和 Buff 等持久事实始终也存在 state 中。

### 位置同步

本地玩家每约 220ms 上报一次遥测，并只在横移方向变化时发送 `move`。房主信任同场聚会玩家声明的位置，只检查数值有效、规范化环绕坐标并保存公开镜像。

普通 snapshot 中本人的 `x/y/vy` 永远不会用于纠正本地物理。每名玩家另有递增的 `positionEpoch`：

- 开局时递增并应用一次房主出生点；
- 每次复活时递增并应用一次房主复活点；
- epoch 不变时，无论网络位置误差多大都不移动本人。

远端玩家位置使用插值显示。因为没有玩家实体碰撞，插值不会参与本地平台物理。

`wrapDelta()` 只用于玩家跨边后的远端插值。平台落地、怪物接触、Buff拾取和子弹目标碰撞全部使用当前屏幕侧的直接横向距离。

客户端检测坠入虚空或无护盾的怪物伤害后立即停止本地操作，并设置 `pendingDeath`。该标志会阻止尚未包含死亡结果的旧 snapshot 把玩家提前恢复；房主确认死亡后统一检查全灭并安排复活。

这不是完整的竞技级回滚网络模型。当前目标是 1–4 人休闲聚会，在低消息频率下保持本地操作及时和共同结果一致。

## 游戏生命周期

```text
lobby
  └─ 所有在线玩家准备
      ▼
running
  ├─ 普通/协作区无限攀登
  ├─ 所有人抵达当前 Boss 门
  │   ▼
  │  boss
  │   └─ Boss HP 清零 → running，进入下一生态
  └─ 所有在线成员死亡
      ▼
gameover
  └─ 房主 restart → lobby
```

重开会保留 `bestRun`，清除本局 seed、玩家统计、动态世界结果和 Buff。在线成员回到未准备状态，掉线席位从下一局名单中移除。

最高纪录当前按本局最高世界高度更新，同时记录该局击败的 Boss 数量。纪录属于房间 state，会随房主 Runtime 的快照恢复；它不是跨房间账号排行榜。

## 开发与构建

在仓库根目录安装依赖：

```bash
pnpm install
```

启动 Parti Web 和 Skyward Room 持续构建：

```bash
pnpm room:dev room-skyward
```

类型检查：

```bash
pnpm --filter @parti/room-skyward typecheck
```

Room 正式构建由仓库根构建统一执行，也可以显式提供输出目录：

```bash
PARTI_ROOM_BUILD_OUT_DIR=/tmp/room-skyward \
  pnpm --filter @parti/room-skyward build:room
```

仓库开发工具链应使用项目约定的 Node 22。使用不匹配的系统 Node 版本可能导致 Vite/esbuild 服务进程异常。

## 人工验收清单

### 房间与生命周期

- 1–4 人可以加入，单人可以准备开局。
- 全员准备前不会开始。
- 开局后新玩家被拒绝。
- 非房主掉线后 Boss 集合被阻塞。
- 全员死亡立即结算。
- 只有房主能返回准备阶段。
- 最高纪录在重开后仍显示。

### 移动与镜头

- 键盘、触屏和重力感应均能改变方向。
- 左右穿边连续。
- 自动弹跳可以到达每个模块出口。
- 向低处队友区域坠落时镜头会向下跟随。
- 不可回退的个人进度底边不会随实际镜头下降。
- 低于团队虚空线后正确死亡。

### 战斗与同步

- 怪物只能被权威击败一次。
- 踩踏会重新弹起。
- 其他玩家能看到开火、散弹和威力子弹。
- 远端展示子弹不会造成伤害或重复命中。
- 团队护盾只消耗一次。
- 复活无敌期间普通怪物不能再次击杀玩家。

### 地形与确定性

- 同一 seed、区块、人数得到相同模块和实体 ID。
- 1–4 人都能完成普通和协作模块。
- 接力桥由开关激活并按时关闭。
- Boss 门横贯世界且保留抵达状态。
- 每个 Boss 后生态、难度、虚空线和下一扇门正确推进。

## 已知边界与后续方向

- 当前内容量仍是验证框架的 MVP，地形模块、敌人和 Boss 行为需要继续扩充。
- 普通敌人位置为确定性静态出生点；复杂移动敌人应增加房主时间锚点或动态状态。
- `RegionRecipe`、Encounter 和 Mechanic 目前是代码结构上的概念，尚未全部提取成独立注册表。
- 遥测校验是轻量合理性检查，不是对恶意客户端的完整反作弊系统。
- Parti 当前使用全量 snapshot；若动态实体数量显著增加，需要继续控制活动状态窗口，或等待 Runtime 支持 patch。
- 不支持主机迁移、跨版本 seed 兼容、历史对局回放或账号级排行榜。

这些边界不妨碍继续增加内容，但新增系统必须继续遵守两个核心约束：静态世界由 seed 推导，改变共同结果的动态事实由房主裁决。
